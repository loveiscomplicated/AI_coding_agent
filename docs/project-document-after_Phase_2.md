# 프로젝트 컨텍스트 문서: Multi-Agent Development System — 5단계 오케스트레이터 연결

## 0. 문서 개요 및 변경 이력

본 문서는 AI 에이전트 팀을 활용한 소프트웨어 개발 파이프라인(Multi-Agent Development System)의 **5단계(오케스트레이터 연결)** 설계를 기록한다. Phase 1(회의 인터페이스)과 Phase 2(단일 에이전트 TDD 파이프라인)가 각각 독립적으로 완료된 상태에서, 이 둘을 연결하고 운영에 필요한 인프라를 구축하는 단계이다.

대화에서는 다음 사항이 순차적으로 논의·확정되었다:

1. 연결 방식(느슨한 파일 기반 vs 백엔드 도입) → 가벼운 로컬 백엔드(FastAPI) 도입 확정
2. 아이맥 상시 서버 활용 가능성 검토 → 설계는 실행 환경 독립적으로, 아이맥은 후보로 보류
3. 태스크 초안 생성 흐름, 파이프라인 트리거 방식, Task Report 소비 방식 상세 설계
4. Discord 핫라인 아이디어 → 최소 버전을 5단계-A에 포함
5. 프론트엔드 구현은 멀티 에이전트가 아닌 Claude Code + 사람 직접 작업으로 분리
6. 보고서 체계(Task Report, Weekly Report) 및 시스템 자기 개선 루프 설계
7. 회의 타입 분리(프로젝트 회의 vs 시스템 회의) 및 컨텍스트 주입 전략
8. 태스크 간 의존성 처리: dev 머지 기반 + 트리 문서(PROJECT_STRUCTURE.md) 기반 컨텍스트 전달
9. 5단계 세부 구현 순서(Step 1~6) 확정
10. 첫 실제 프로젝트 선정: 시스템 자체의 유틸리티 모듈(셀프 호스팅 검증)

## 1. 프로젝트 배경 및 현재 상태

### 1.1 전체 시스템 구조 (기존)

```
나 (사람)
 ↕  회의 / 보고서 검토 / PR 승인
중앙 오케스트레이터 (Claude Sonnet)
 ↕  태스크 분배 / 결과 수집 / 컨텍스트 압축
┌──────────────────────────────────┐
│  Agent 1    Agent 2    Agent N   │  ← Claude Haiku
│  [샌드박스]  [샌드박스]  [샌드박스] │
│  테스트 → 코드 → 리뷰 (순차)     │
└──────────────────────────────────┘
 ↓  PR 생성 + 승인 대기
실제 코드베이스 (Git)
```

### 1.2 완료된 Phase

**Phase 1 (회의 인터페이스) ✅**: React SPA (Vite + Tailwind), Anthropic API 직접 호출(`dangerouslyAllowBrowser: true`), 마크다운 컨텍스트 문서 생성, 이전 회의 로드, 파일/이미지 첨부, 선택지 버튼 등 완성.

**Phase 2 (단일 에이전트 TDD 파이프라인) ✅**: Docker 테스트 러너, ScopedReactLoop, TestWriter/Implementer/Reviewer 에이전트, TDDPipeline 상태 머신, GitWorkflow(브랜치→커밋→PR), YAML 기반 태스크 관리. E2E 검증 완료(정수 계산기, 단어 빈도 분석기).

### 1.3 현재 문제: 두 시스템의 단절

```
Phase 1: 회의 UI (React) → context_doc.md 생성
Phase 2: 파이프라인 (Python CLI) → tasks.yaml 읽어서 실행
                  ↑
             수동으로 연결 (사람이 직접 tasks.yaml 작성)
```

5단계의 핵심 과제는 이 단절을 해소하는 것이다.

## 2. 5단계 핵심 설계 결정

### 2.1 연결 방식: FastAPI 로컬 백엔드 도입

**검토된 방향:**

| 방향 | 설명 | 판단 |
|------|------|------|
| A: 완전 파일 기반 | context_doc 다운로드 → 터미널에서 스크립트 실행 → 보고서 드래그&드롭 | **폐기** — 수동 워크플로우 문서화에 불과, 실질적 연결 아님 |
| B: 가벼운 로컬 백엔드 | FastAPI로 얇은 서버, 프론트엔드가 이 서버와 통신 | **채택** |
| C: 통합 백엔드 (풀스택) | WebSocket, 실시간 모니터링, 인증 등 완비 | **시기상조** — Phase 3에서 확장 |

**방향 B 채택 이유:**
1. **Phase 3 필연성**: 병렬 에이전트, 실시간 모니터링, 보고서 자동화 — 프론트엔드만으로 불가능. 지금 가벼운 백엔드를 도입하면 Phase 3에서 확장만 하면 됨.
2. **보안 개선**: `dangerouslyAllowBrowser: true`로 프론트엔드에서 직접 API 키를 다루는 현 구조를 탈피. 백엔드가 API 키를 관리.
3. **회의 → 태스크 → 실행 → 보고서** 사이클이 하나의 UI 안에서 이어지는 실질적 오케스트레이터 연결.

### 2.2 실행 환경 독립성

**설계 원칙**: 5단계 설계는 "어디서 돌리느냐"에 독립적으로 만든다. FastAPI 백엔드는 맥북 로컬, 아이맥 상시 서버, 클라우드 VPS 어디서든 동일하게 동작해야 한다.

**아이맥 상시 서버 후보 검토:**

- 2015 Late iMac 21.5" (Intel Broadwell i5-5250U or i7-5775R, 8GB LPDDR3, HDD 1TB)
- 최대 macOS Monterey(12) 지원 — Docker Desktop 경계선, 향후 지원 끊길 가능성
- API 호출 기반 시스템이므로 로컬 연산 부하는 미미 → 하드웨어 자체는 충분
- **주요 우려**: HDD 기반 모델이면 파일 I/O 병목, Docker 호환성 불안정, macOS 보안 패치 종료
- **대안**: venv 기반 격리로 Docker 대체 가능, Colima 경량 런타임 시도 가능
- **결론**: 가능성 열어두되, 실측 확인 필요. 5단계 설계 자체는 환경 무관하게 진행.

### 2.3 프론트엔드 구현의 특수 처리

**결정: 프론트엔드는 멀티 에이전트 시스템이 구현하지 않고, Claude Code + 사람이 직접 구현한다.**

**배경:**
- 프론트엔드는 "좋아 보이는가"가 판단 기준의 상당 부분을 차지 → TDD로 검증 불가
- 레이아웃, 인터랙션, 간격 등 감각적 피드백이 필수 → 사람이 눈으로 보면서 실시간 수정해야 함
- Claude Code의 즉각적 피드백 루프(코드 수정 → 눈으로 확인 → 지시)가 프론트엔드에 최적

**역할 분담:**
```
멀티 에이전트 시스템:
├── 백엔드 구현 (자율 TDD)
└── 프론트엔드 구현 컨텍스트 문서 생성
    ├── 컴포넌트 구조/계층
    ├── 각 컴포넌트의 역할과 props 인터페이스
    ├── 백엔드 API 스펙 (프론트가 소비할)
    ├── 상태 관리 설계
    └── 디자인 가이드라인/참고 이미지

Claude Code + 사람:
└── 위 컨텍스트 문서를 받아서 프론트엔드 구현
```

**장점**: 백엔드가 먼저 완성되고 API 스펙이 확정된 상태에서 프론트를 만드는 흐름이 자연스럽게 강제 → 인터페이스 불일치 문제 감소.

**컨텍스트 문서 적정 수준**: "설계 수준의 구체성" — 컴포넌트 트리, API 호출 매핑, 주요 상태 흐름. 너무 추상적이면 쓸모없고, 코드 수준까지 내려가면 에이전트가 직접 구현하는 것과 동일.

### 2.4 태스크 타입 분기 (5단계-B로 연기)

tasks.yaml에 `type: backend | frontend` 필드를 추가하여, 타입별로 파이프라인 동작을 분기하는 기능. backend은 기존 TDD 파이프라인, frontend은 컨텍스트 문서 생성만 수행. 5단계-A에서 기본 흐름이 안정된 후 추가.

## 3. 태스크 초안 생성 흐름

### 3.1 변환 흐름

```
회의에서 context_doc.md 확정
  → "태스크 생성" 요청 (UI 또는 API)
  → Sonnet이 context_doc.md를 읽고 tasks.yaml 초안 생성
  → 오케스트레이터가 의존성 순서 자동 계산
  → UI에서 태스크 목록 확인 / 수정 / 삭제 / 추가
  → 사람 승인
  → 파이프라인 실행 시작
```

### 3.2 Sonnet의 판단 범위

**Sonnet이 해도 되는 것:**
- 기능을 적절한 크기의 태스크로 분할
- acceptance_criteria를 테스트 가능한 형태로 구체화
- 태스크 간 의존 순서 제안 (depends_on 필드)

**Sonnet이 하면 안 되는 것:**
- 회의에서 결정하지 않은 기능을 임의로 추가
- 기술 스택이나 아키텍처를 변경하는 판단

### 3.3 태스크 크기 가이드라인

**"하나의 태스크는 하나의 Python 모듈(파일 1~2개) 수준"** — Phase 2 E2E 검증의 정수 계산기/단어 빈도 분석기 수준. Haiku가 안정적으로 구현할 수 있는 크기를 유지. 이 가이드라인은 Sonnet의 태스크 생성 프롬프트에 포함.

### 3.4 tasks.yaml 확장 형식

```yaml
tasks:
  - id: "001"
    title: "메트릭 수집기"
    type: backend              # 5단계-B에서 활용
    description: "..."
    depends_on: []             # 의존하는 태스크 ID 리스트
    acceptance_criteria:
      - "..."
    status: pending
```

## 4. 파이프라인 트리거 및 실행 방식

### 4.1 기본 동작: 배치 실행 + 핫라인 개입

```
tasks.yaml 초안 승인
  → 파이프라인이 의존성 순서에 따라 순차 실행
  → 각 태스크 완료마다 핫라인(Discord) 알림
  → 사람이 핫라인으로 "멈춰" → 다음 태스크부터 일시정지
  → 사람이 "계속" → 재개
```

**설계 근거**: 태스크마다 개별 승인하면 사람 시간을 너무 잡아먹어 에이전트 활용 의미가 퇴색. 무조건 배치로 돌리면 잘못된 방향을 늦게 발견. **기본은 배치, 핫라인으로 언제든 개입 가능**이 균형점.

### 4.2 태스크 간 의존성 처리

**세 가지 문제와 각각의 해결:**

| 문제 | 해결 방식 |
|------|-----------|
| 실행 순서 | 오케스트레이터(Sonnet)가 의존성 순서 자동 계산 (위상 정렬) |
| 코드 접근 | task-001 완료 → dev 브랜치에 머지 → task-002는 dev 기반 workspace 생성 |
| 컨텍스트 전달 | PROJECT_STRUCTURE.md (트리 문서) 기반 — 아래 4.3 참조 |

### 4.3 컨텍스트 전달: 트리 문서(PROJECT_STRUCTURE.md)

**검토된 두 가지 방법과 결론:**

**방법 1: 커밋 메시지 + docstring 표준화**
- 강점: 코드 자체에 정보 내장, 별도 관리 불필요
- 약점: 에이전트가 관련 파일을 찾아 열어봐야 함 → 토큰/시간 낭비
- **결론: 코드 품질 표준으로 강제하되, 컨텍스트 전달 수단으로는 불충분**

**방법 2: 프로젝트 구조 트리 문서 (PROJECT_STRUCTURE.md)**
- 강점: 한 문서로 전체 코드베이스 구조/인터페이스 즉시 파악. 에이전트와 사람 모두에게 유용.
- 약점: 코드와 불일치 위험 → StructureUpdater 단계로 해결
- **결론: 컨텍스트 전달의 메인 수단으로 채택**

**최종 결정: 둘 다 사용하되 역할을 분리**
- 방법 1 (docstring): 코드 품질 기준. 모든 에이전트 프롬프트에 docstring 규칙 포함.
- 방법 2 (트리 문서): 컨텍스트 전달의 메인 수단. 에이전트가 태스크 시작 시 첫 번째로 읽는 문서.

**트리 문서 파싱 깊이: 레벨 3**
- 파일명 + 클래스/함수명 + 함수 시그니처(인자, 리턴 타입) + docstring 첫 줄 요약
- 레벨 2(시그니처만)는 함수 역할 파악을 위해 결국 파일을 열어야 하므로 불충분
- 레벨 4(전체 docstring)는 문서 비대화로 토큰 낭비

**불일치 방지: StructureUpdater 파이프라인 단계 추가**

```
TDD 파이프라인 (수정):
  TestWriter → Implementer → Reviewer → StructureUpdater → PR
```

StructureUpdater는 LLM이 아닌 **Python AST 파싱 스크립트**를 실행하여 실제 코드에서 트리 문서를 자동 생성. 코드에서 직접 생성하므로 불일치가 구조적으로 불가능.

## 5. Discord 핫라인

### 5.1 핵심 동기

**"언제 어디서든 멀티 에이전트 시스템과 연결"** — 단순 편의성이 아니라 시스템과의 관계 자체를 바꾸는 핵심 인터페이스.

```
핫라인 없이: 맥북 앞에 앉아서 → 파이프라인 실행 → 결과 대기 → 확인
핫라인 있으면: 태스크 등록 → 외출 → 폰으로 진행상황 알림 → 
             필요시 답변 → 돌아와서 PR 확인
```

비동기 감독이 가능해져 워크플로우가 근본적으로 달라짐.

### 5.2 Discord 선택 이유

- Discord Bot API가 잘 되어 있음
- 채널별 프로젝트/태스크 분리 가능
- 모바일에서 알림 + 즉시 답장
- 대화 기록 자연스럽게 보존

### 5.3 핫라인 범위 설계

**오케스트레이터가 물어봐도 되는 것**: 비즈니스 로직 판단, 우선순위 변경, 에러 처리 정책
**물어보면 안 되는 것**: 코드 스타일, 변수명, 구현 세부사항

### 5.4 5단계-A: 최소 버전

```
├── 파이프라인 시작/완료 알림
├── 오케스트레이터 질문 → 디스코드 메시지 전송
├── 내 텍스트 답변 → 파이프라인에 전달
└── 끝. 텍스트 메시지 양방향만.
```

예시:
```
[봇] 🚀 task-001 "사용자 인증 API" 파이프라인 시작
[봇] ❓ task-001 질문: 인증 실패 시 429 vs 401?
[나] 401로 해
[봇] ✅ task-001 완료. PR #12 생성됨
```

### 5.5 5단계-B: 확장 버전 (후순위)

| 기능 | 설명 |
|------|------|
| 태스크 승인/거부 | 디스코드에서 버튼으로 직접 승인 |
| PR 요약 전송 | 변경 파일, 주요 내용 요약을 디스코드에 자동 게시 |
| 스크린샷 피드백 | 프론트엔드 태스크에서 시각적 피드백 |
| 파이프라인 트리거 | `/run task-001 task-002` 명령어로 디스코드에서 실행 |

확장 버전은 버튼/인터랙션 컴포넌트, 명령어 파서, 권한 관리 등 구현량이 상당히 늘어나므로 기본 흐름 안정화 후 추가.

### 5.6 타임아웃 정책

**결정: 지금은 규칙을 정밀하게 설계하지 않는다.**

- **단일 태스크 실행 중(현재)**: 질문 발생 시 파이프라인 일시정지, 답할 때까지 기다림. 하나만 돌고 있으니 손해 없음.
- **병렬 에이전트(Phase 3)**: 질문 발생한 태스크만 일시정지, 의존성 없는 다른 태스크는 계속 진행.
- **타임아웃 값**: Phase 3에서 실제 패턴(질문 빈도, 유형, 응답 시간)을 관찰한 후 결정.

## 6. 보고서 체계

### 6.1 Task Report

**생성 시점**: 태스크 완료 시 자동 생성
**저장 형식**: YAML (`data/reports/task-{id}.yaml`)
**소비자**: 오케스트레이터(상세), 사람(핫라인 요약)

**구조:**
```yaml
task_id: "001"
title: "..."
status: COMPLETED  # or FAILED
completed_at: "2026-04-07T14:30:00Z"
metrics:
  retry_count: 2
  total_tokens: 12450
  cost_usd: 0.52
  test_count: 7
  test_pass_first_try: false
  reviewer_verdict: APPROVED
  time_elapsed_seconds: 180
  failure_reasons: ["TypeError in line 23", "missing edge case"]
pipeline_result:
  test_output_summary: "7 passed in 1.2s"
  reviewer_feedback: "..."
  pr_number: 12
  branch: "agent/task-001"
```

**구조화된 메트릭의 중요성**: 이 데이터가 시스템 자기 개선 루프의 기반이 됨. 나중에 분석하고 싶어도 데이터가 없으면 불가능하므로, 처음부터 구조화하여 저장.

### 6.2 Weekly Report

**생성 주기**: 매주 (Daily Summary는 핫라인 알림으로 대체)
**저장 위치**: `data/reports/weekly/2026-W15.md`

**Daily Summary를 폐기한 이유**: 핫라인으로 태스크별 알림이 이미 오므로, Daily Summary의 단순 나열은 중복. Weekly Report가 데이터 축적 후 의미 있는 분석을 제공하는 최소 주기.

**핵심 섹션:**
1. 진행 요약 (완료/실패/보류)
2. 주요 성과
3. **이슈 & 패턴** — 단순 숫자가 아닌 시스템 개선 포인트 제안 (예: "재시도율 높음 → criteria 강화 필요")
4. **비용** — 모델별 비용, 합계, 태스크당 평균
5. **추이** — 전주 대비 주요 지표 증감
6. 다음 주 계획

**핵심 지표: 태스크당 평균 비용**. 시간에 따른 추이가 시스템 성숙도의 핵심 신호. 감소 → 시스템 개선, 증가 → 비효율 발생.

### 6.3 Monthly Report

현재 미구현. 프로젝트 규모가 커져서 주간 데이터만으로 불충분할 때 추가.

### 6.4 컨텍스트 압축 (미구현, 필요 시 추가)

기존 설계의 `Task Report → Daily → Milestone` 계층적 압축은 5단계에서 바로 구현하지 않음. 단일 에이전트에 태스크도 적어 컨텍스트 오버플로우가 빨리 발생하지 않을 것으로 판단. 실제 문제가 될 때 추가.

## 7. 회의 인터페이스 확장

### 7.1 회의 타입 분리

**프로젝트 회의**: "무엇을 만들 것인가"
- 특정 프로젝트의 기능, 설계, 우선순위 논의
- context_doc + execution_brief + PROJECT_STRUCTURE.md 주입
- 프로젝트마다 별도 존재

**시스템 회의**: "어떻게 더 잘 만들 것인가"
- 멀티 에이전트 시스템 자체의 성능, 비용, 프로세스 개선
- 전체 프로젝트 통합 메트릭 + 패턴 분석 주입
- 시스템 하나에 대해 하나
- **주기: 매주** (Weekly Report 생성 직후가 적절)

**시스템 회의 안건 예시:**
- 비용 분석 (프로젝트별, 태스크 유형별, 모델 선택 재검토)
- 품질 패턴 (재시도율 추이, CHANGES_REQUESTED 빈도, 실패 유형)
- 프로세스 개선 (핫라인 질문 패턴 → 자동화 가능성, criteria 품질과 성공률 상관관계)
- 시스템 확장 논의 (새 모델 검토, 도구 추가, Phase 3 준비도)

**회의 타입을 분리하는 이유:** 프로젝트 회의에서 비용/시스템 이야기가 나오면 프로젝트 방향 논의를 방해. 시스템 회의를 프로젝트 회의에서 하면 특정 프로젝트에 편향된 최적화 위험.

**UI 반영:**
```
회의 목록 UI:
├── [프로젝트] AI 코딩 에이전트 — 3월 30일
├── [프로젝트] 사이드 프로젝트 X — 4월 2일
├── [시스템] 멀티 에이전트 개선 — 4월 5일
```

### 7.2 회의 시작 시 컨텍스트 주입 전략

**기존 문제**: Phase 1의 "이전 회의 로드"는 대화 히스토리를 통째로 로드. 파이프라인 결과는 대화 밖에서 일어난 일이라 히스토리에 끼워넣을 수 없음.

**해결: 두 가지 회의 모드와 컨텍스트 분리**

```
"이전 회의 이어서" (기존 유지):
  → 대화 히스토리 통째로 로드
  → 단기 연속 작업용 (어제 회의 이어서)

"주간 회의 시작" (신규):
  → 자동 주입 (요약):
    ├── context_doc.md (이전 회의 결정사항)
    ├── execution_brief.md (파이프라인 결과 요약) ← 이 시점에 생성
    └── PROJECT_STRUCTURE.md (현재 코드 구조)
  → 이전 대화 히스토리는 로드하지 않음
  → Opus에게 필요한 건 결정의 결과이지, 결정 과정 전체가 아님
```

**execution_brief 생성 시점**: "주간 회의 시작" 버튼 클릭 시 즉시 생성. 마지막 회의 이후의 Task Report들을 백엔드가 수집 → Sonnet이 요약 → 회의 시작 전에 Opus 시스템 프롬프트에 주입.

```
"주간 회의 시작" 버튼 클릭
  → 백엔드가 마지막 회의 이후의 Task Report들 수집
  → Sonnet이 execution_brief 생성
  → context_doc + execution_brief + PROJECT_STRUCTURE.md 주입
  → 회의 시작
  → Opus: "지난주 8개 태스크 중 7개 완료했고, task-008은 
          DB 스키마 불일치로 실패했어요. 이 부분부터 논의할까요?"
```

**온디맨드 상세 참조**: 회의 중 "task-008 자세히 봐줘" → Task Report 상세 로드. "auth 모듈 코드 보여줘" → 실제 코드 로드.

**시나리오 3 (긴급 회의)**: "주간 회의 시작" + 온디맨드 상세 리포트 조합으로 대응. 별도 구현은 하되, 기존 두 모드의 조합으로 커버 가능.

### 7.3 주간 루틴 예시

```
월요일: 프로젝트 회의 → tasks.yaml 생성 → 승인 → 파이프라인 시작
화~금: 파이프라인 실행, 핫라인으로 알림/질의응답
금요일: Weekly Report 자동 생성
토/일: 시스템 회의 → 프로세스 개선 결정 → 다음 주 프로젝트 회의에 반영
```

## 10. 5단계-A 구현 순서

```
Step 1: FastAPI 백엔드 + API 프록시
  → 프론트엔드가 백엔드 경유하도록 전환
  → dangerouslyAllowBrowser 제거
  → 기존 Phase 1이 백엔드 위에서 동작 확인

Step 2: 파이프라인 확장
  → StructureUpdater 추가
  → dev 머지 후 다음 태스크 workspace 생성
  → Task Report + 메트릭 자동 저장
  → 파이프라인 실행/상태 API 연결

Step 3: 태스크 초안 생성
  → context_doc → Sonnet → tasks.yaml 변환
  → 의존성 자동 계산 (위상 정렬)
  → UI에서 태스크 목록 확인/수정/승인

Step 4: 회의 인터페이스 확장
  → 회의 타입 (프로젝트/시스템) 분리
  → execution_brief 자동 생성 + 주입
  → 온디맨드 상세 리포트 로드

Step 5: Discord 핫라인
  → 봇 기본 세팅
  → 파이프라인 알림 연결
  → 질의응답 흐름 연결

Step 6: 보고서 체계
  → Weekly Report 자동 생성
  → 시스템 회의용 메트릭 집계
```

**순서 설계 근거:**
- Step 1(백엔드)이 나머지 전부의 기반 → 최우선
- Step 2(파이프라인 확장)를 Step 3(태스크 생성) 앞에 둔 이유: 기존 Phase 2 코드 개선이라 익숙한 영역, 불확실성 낮음. 익숙한 걸 먼저 안정화 후 새 기능 추가.
- Step 5(Discord)를 뒤로 뺀 이유: 핫라인 없이도 Step 1~4가 웹 UI에서 동작. 핫라인은 편의성 레이어이지 기능적 의존성 아님. 최종 시스템에는 필수이지만 구현 순서상 먼저일 필요 없음.

### 10.1 5단계 구현 주체

**5단계 인프라 자체는 Claude Code + 사람이 직접 구현.** 멀티 에이전트 시스템이 아직 미완성인 상태에서 자기 자신을 만드는 것은 닭과 달걀 문제. 유틸리티 모듈 프로젝트는 시스템 검증용 테스트 프로젝트로 활용.

## 11. 첫 실제 프로젝트: 유틸리티 모듈 (셀프 호스팅 검증)

### 11.1 프로젝트 선정 과정

**검토 후 보류: SNN 연구 프로젝트**
- GPU 훈련 필요 (Docker 샌드박스에서 CUDA 실행 불가)
- 하이퍼파라미터 탐색이 핵심 (코드 작성보다 실험 설계)
- 기존 코드베이스 확장 (새로 만드는 것이 아님)
- 결과 해석에 주관적 판단 필요
- → TDD 파이프라인과 미스매치 → SNN은 시스템 안정화 후 진행

**채택: 시스템 자체의 유틸리티 모듈**
- 순수 Python, 명확한 입출력, 테스트 용이
- 만들고 나면 바로 시스템에 통합
- 모듈 간 의존성이 적당히 존재 → 의존성 처리 검증
- "시스템이 자기 자신의 부품을 만든다" = 강력한 셀프 호스팅 검증

### 11.2 대상 모듈 (5개)

1. **메트릭 수집기** (`metrics/collector.py`) — Task Report 저장/로드/집계
2. **Weekly Report 생성기** (`reports/weekly.py`) — 주간 마크다운 보고서 + 패턴 분석
3. **PROJECT_STRUCTURE.md 생성기** (`structure/updater.py`) — Python AST 파싱 → 트리 문서
4. **execution_brief 생성기** (`reports/execution_brief.py`) — 회의 시작 시 주입할 실행 요약
5. **태스크 의존성 계산기** (`orchestrator/dependency.py`) — 위상 정렬 기반 실행 순서 결정

### 11.3 모듈 간 의존성

```
task-001: 메트릭 수집기        (의존 없음)
task-003: 구조 생성기          (의존 없음)
task-005: 의존성 계산기        (의존 없음)
    ↓
task-002: Weekly Report 생성기  (→ task-001)
task-004: execution_brief 생성기 (→ task-001)
```

실행 순서: `001 → 003 → 005 → 002 → 004` (그룹 1 내 순서는 임의)

**셀프 레퍼런스**: 이 의존성 구조 자체가 task-005 (의존성 계산기)의 테스트 케이스로 활용 가능.

### 11.4 별도 컨텍스트 문서

유틸리티 모듈 프로젝트의 상세 컨텍스트 문서는 별도로 작성 완료 (completeness: 78). 각 모듈의 인터페이스, 데이터 형식, 디렉토리 구조, 파이프라인 통합 계획, 검증 전략 포함.
